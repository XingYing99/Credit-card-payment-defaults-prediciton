The Data Modeling Problem¶

This project aims to find the best classification model to predict credit card payment defaults using a given data set, which contains payment information of 30,000 credit card holders obtained from a bank in Taiwan. Each data sample is described by 23 feature attributes. The target feature (Attribute Y) to be predicted is a binary variable, 0 (= not default) or 1 (= default). We will be evaluating the models based on their ability to correctly predict clients that will and will not default on their payment next month. This is because it is equally important to predict both default and non-default cases as predicting the former would allow financial institutions to avoid financial losses to potential defaulters while predicting the latter would mean that credit applications will not be falsely rejected, compromising potential sales. Thus, the main metric for evaluation will be the average accuracy due to the skewness of the target variable and the desire to predict whether the person defaults or doesn't default correctly.

Data Pre-processing¶

Data pre-processing involves transforming raw data into an understandable format. As the raw data may have several anomalies, there is a need for data to be well prepared so that accurate and effective models can be built. We have attempted to pre-process the dataset through the steps below to get a sense of whether they were helpful for our models. For the purpose of our research, we have decided to proceed with 3 levels of data pre-processing (minimal, mid, intensive).

For minimal-level pre-processing, all we did was make all of the PAY_i variables binary. This means we replaced each of these variables with 0 if the individual paid late (e.g. PAY_0 > 0), and replaced this variable with 1 if the individual paid on time.

Next, for mid-level preprocessing, we built upon the minimal-level pre-processing and added ordinal dummies for education. There are 6 dummies for education, each dummy a binary variable for that level of education. We are assuming that an individual with a high educational level would have completed the levels before it. For example, an individual with a graduate school education would have 1 for all the dummy variables, indicating that this individual has completed all levels of education up to and including graduate school.

Finally for our intensive-level preprocessing, we built upon the mid-level pre-processing and normalized most of the data. We chose to normalize the data because we did not want the difference in the range of the values for each variable to introduce bias to our model, such as distributing more weight to larger values even though these variables might not actually be more effective in predicting a default. Before we proceeded with the normalization (during exploratory analysis), we plotted the histograms for each of our continuous variables to inspect their distributions. Since all of our variables did not look to be a normal distribution, we did not opt for the z-index standardization and instead proceeded with the min-max normalization.

In addition, we made variables such as sex and marriage binary, where sex=1 if an individual is male and sex=0 if an individual is female. Marriage=1 if an individual is married and marriage=0 if an individual is single or others.

Lastly, we chose to down sample our training data upon realization that there is an imbalance in the proportion of default=1 and default=0. More specifically, we have 7852 individuals that have default=0 and only 2148 individuals with default=1. This means that the training model will spend most of its time on the default=0 instead of default=1, which are essentially the individuals that we should worry about.

However, upon training each of these pre-processed data with our first model, which was a logistic model, we found that the accuracy using the intensive-level pre-processing was very low compared to the accuracy from a minimally preprocessed data. On the other hand, the difference in accuracy between our min and mid preprocessed data is negligible. As mentioned previously, our mid level preprocessed data includes the ordinal education dummies. Including these dummies will allow us to analyze the education variable at each independent level, which permits the t-test for significance. Therefore, since the difference in accuracy is negligible, we have opted to proceed with our other models with only the mid level preprocessed data.

Since the raw dataset is large, it may be appropriate to reduce the size to improve the efficiency of the machine learning algorithms. This can be done through feature selection, whereby irrelevant features are eliminated.

# evaluation values
avg_accuracy <- function(actual, preds){
    if (!is.numeric(actual)){
        actual <- as.numeric(actual)-1
    }
    if(!is.numeric(preds)){
        preds <- as.numeric(preds)-1
    }
    tn <- sum((actual == preds) & (actual == 0))
    tp <- sum((actual == preds) & (actual == 1))
    all_ones <- sum(actual)
    all_zeros <- sum(!actual)
    
    avg_acc <- (tn/all_zeros + tp/all_ones)/2
}
eval_res <- function(actual, preds){
    acc <- Accuracy(preds, actual)
    auroc <- AUC(preds, actual)
    recall <- Recall(actual, preds, positive = "1")
    precision <- Precision(actual, preds, positive = "1")
    #sens <- Sensitivity(actual, preds, positive = "1")
    avg_acc <- avg_accuracy(actual, preds)
    spec <- Specificity(actual, preds, positive = "1")
    
    c(acc, precision, recall, avg_acc, spec, auroc)
}

Random Forest¶

Four different methods for feature selection for the random forest model were tested to select the best features: using the random forest model, chi squared, t-test, and the p-value of the chi squared results. The top 8, 10, and 15 features of each of these methods was selected and used to train the random forest model. The accuracy and area under the ROC was calculated for each method. The method which performed the best was used in the model selection process. Table X shows that the random forest model using the top 15 features ranked by importance from the random forest method performed the best. In the random forest model, using all of the features isnt always the best approach since not all features may be used in the actual final model. get_feat_info <- function(data_used, test_data){
  cols <- names(data_used)
  #features2 <- cols[cols != 'ID']
  features <- cols[cols != 'default_payment_next_month']
  feat_list <- features
  

  
  rf <- randomForest(as.factor(default_payment_next_month)~., data_used)

  
  feats <- importance(rf)
  
  rf_vals <- feats[, 1]
  
  
  issue_chi <- tibble(features = features, p_val=1:length(features), stat = 1:length(features),
                      ttest = 1:length(features),
                      rf = rf_vals)
  count <- 1
  for (i in features){
    tab <- table(as.factor(data_used[['default_payment_next_month']]), data_used[[i]])
    chi <- chisq.test(tab)
    ttest <- t.test(tab)
    issue_chi$p_val[issue_chi$p_val == count] <-chi$p.value
    issue_chi$stat[issue_chi$stat == count] <-chi$statistic
    issue_chi$ttest[issue_chi$ttest == count] <-ttest$statistic[1]
    
    count <- count + 1
  }
  
  issue_chi
}

Future Improvements¶

There are some rooms for improvement that we have considered:

  • There were many more class 0 data than class 1 data in the dataset which meant that the accuracy of our models could have been skewed towards correct prediction of class 0 data. This means that we could have used harmonic mean to calculate accuracy so as to mitigate bias towards class 0 prediction.
  • Instead of splitting the data into training and test data, if more data was given, we would be able to split our training data further into training and validation data set. This is so that when we fit a model on the training data, we can then use the fitted model to predict responses in the validation data, and tune the features in the model, before moving on to the test data.
